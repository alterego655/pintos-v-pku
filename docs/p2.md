#### ALGORITHMS

>A2: Briefly describe how you implemented argument parsing.  How do you arrange for the elements of argv[] to be in the right order?
>How do you avoid overflowing the stack page?

**Argument Parsing Implementation:**

1. **Tokenization**: Use `strtok_r()` to split the command line string on spaces, storing each token in a temporary `argv[]` array and incrementing `argc`.

2. **Stack Layout**: Push arguments onto the stack in reverse order (argc-1 down to 0) so they appear in correct order when accessed. Each string is copied to the stack and its stack address is stored in `argv[i]`.

3. **Stack Alignment**: Align the stack pointer to 4-byte boundaries using `ALIGN_DOWN()` for proper word access.

4. **Standard Format**: Push in order: null sentinel, argv pointers (reverse order), argv pointer, argc, fake return address.

**Overflow Prevention:**
- Track `total_len` of all arguments and fail if exceeds `PGSIZE`
- Limit `argc` to 127 to prevent array overflow
- Validate each step doesn't exceed stack page boundaries

#### RATIONALE

>A3: Why does Pintos implement strtok_r() but not strtok()?

`strtok_r()` is reentrant (thread-safe) while `strtok()` uses static internal state. In a multi-threaded kernel environment, multiple threads could simultaneously parse strings. `strtok()` would cause race conditions as threads would interfere with each other's parsing state. `strtok_r()` requires the caller to provide the state variable (`save_ptr`), making it safe for concurrent use.

>B2: Describe how file descriptors are associated with open files. Are file descriptors unique within the entire OS or just within a single process?

File descriptors are unique **within a single process**, not system-wide. Each thread has its own `fd_table[FD_CNT]` array where:
- Index 0 and 1 are reserved for stdin and stdout
- Indices 2-127 are available for regular files
- Each entry contains a pointer to a `struct file` or NULL if closed
- When opening a file, we scan for the first available slot and return its index
- The same file can have different FD numbers in different processes

#### ALGORITHMS

>B3: Describe your code for reading and writing user data from the kernel.

**Memory Validation Strategy:**
I use a comprehensive validation-then-access approach with multiple layers:

1. **Pointer Validation** (`is_valid_user_ptr`):
   - Check if pointer is NULL or not in user virtual address space
   - Use `pagedir_get_page()` to verify the page is mapped in user's page directory

2. **Range Validation** (`is_valid_read_range`):
   - Check for pointer arithmetic overflow
   - Verify end address is still in user space
   - Validate every byte in the range is mapped

3. **Safe Copying** (`copy_from_user`/`copy_to_user`):
   - Validate both source and destination addresses
   - Copy byte-by-byte with validation at each step
   - Return false immediately if any validation fails

4. **System Call Argument Extraction** (`get_syscall_arg`):
   - Calculate argument address from stack pointer
   - Use `copy_from_user` to safely read argument
   - Terminate process if extraction fails

**Example Flow for write() syscall:**
1. Extract fd, buffer, size using `get_syscall_arg()`
2. Validate entire buffer range with `is_valid_read_range()`
3. Proceed with write operation using validated buffer

**Implementation using get_user()/put_user():**
```c
static int get_user (const uint8_t *uaddr) {
  int result;
  asm ("movl $1f, %0; movzbl %1, %0; 1:"
       : "=&a" (result) : "m" (*uaddr));
  return result;  // Returns -1 on fault, byte value on success
}

static bool put_user (uint8_t *udst, uint8_t byte) {
  int error_code;
  asm ("movl $1f, %0; movb %b2, %1; 1:"
       : "=&a" (error_code), "=m" (*udst) : "q" (byte));
  return error_code != -1;
}
```

**Key Advantage:** This approach assumes user pointers are usually valid (optimistic) and only pays the cost when access actually fails, whereas my current approach assumes they might be invalid (pessimistic) and always pays validation cost upfront.

**Performance Comparison:**
- **My approach**: 4,096 page table lookups + 4,096 memory writes = 8,192 operations
- **Fault approach**: 4,096 memory writes + 0 page faults = 4,096 operations (50% faster)

>B5: Briefly describe your implementation of the "wait" system call
>and how it interacts with process termination.

**Wait System Call Implementation:**

1. **Child Lookup**: Find the child's `child_status` record in parent's children list using `child_status_find()`

2. **Validation**: Return -1 if:
   - Child TID not found (not a child)
   - Parent already called wait() on this child
   - Child TID is TID_ERROR (invalid)

3. **Synchronization**: 
   - If child hasn't exited yet, block on `exit_sema` until child signals completion
   - Child signals `exit_sema` in `process_exit()` when it terminates

4. **Cleanup**: 
   - Mark child as waited upon to prevent double-wait
   - Remove from children list and free `child_status` record
   - Return child's exit code

**Process Termination Interaction:**
- When process exits, it sets `child_exited = true` and signals `exit_sema`
- Parent can be waiting (blocked on semaphore) or wait later (finds child already exited)
- Exit code is preserved in `child_status` until parent waits or exits

>B6: Any access to user program memory at a user-specified address
>can fail due to a bad pointer value.  Such accesses must cause the
>process to be terminated.  System calls are fraught with such
>accesses, e.g. a "write" system call requires reading the system
>call number from the user stack, then each of the call's three
>arguments, then an arbitrary amount of user memory, and any of
>these can fail at any point.  This poses a design and
>error-handling problem: how do you best avoid obscuring the primary
>function of code in a morass of error-handling?  Furthermore, when
>an error is detected, how do you ensure that all temporarily
>allocated resources (locks, buffers, etc.) are freed?  In a few
>paragraphs, describe the strategy or strategies you adopted for
>managing these issues.  Give an example.

**Strategy for Clean Error Handling:**

1. **Fail-Fast Validation**: I validate all user memory accesses at the beginning of each system call before acquiring any resources. This prevents the need to unwind complex state.

2. **Centralized Validation Functions**: I created reusable validation functions (`is_valid_user_ptr`, `is_valid_read_range`, `get_syscall_arg`) that encapsulate all validation logic, keeping the main syscall handlers focused on their primary function.

3. **Early Return Pattern**: System calls return immediately on validation failure by calling `thread_exit()`, which automatically handles cleanup through the normal process exit path.

4. **Resource Management**: I use RAII-style patterns where resources are acquired late and cleaned up automatically:
   - Filesystem lock is acquired only when needed and released in same function
   - File descriptors are managed through the process exit cleanup
   - Memory allocations are minimal and freed through normal exit paths

**Example - write() syscall:**
```c
// 1. Validate and extract arguments first (fail fast)
if (!get_syscall_arg(f, 1, &fd, sizeof(int)) ||
    !get_syscall_arg(f, 2, &buf, sizeof(char *)) ||
    !get_syscall_arg(f, 3, &size, sizeof(size_t)))
  thread_exit();  // No resources to clean up yet

// 2. Validate user buffer before doing anything else
if (!is_valid_read_range(buf, size))
  thread_exit();  // Still no resources acquired

// 3. Only now proceed with main function
if (fd == 1) {
  // ... putbuf logic
} else {
  fs_lock_acquire();  // Acquire resources late
  // ... file operations
  fs_lock_release();  // Release immediately
}
```

This approach ensures that either the syscall succeeds completely or fails cleanly with no resource leaks.

#### SYNCHRONIZATION

>B7: The "exec" system call returns -1 if loading the new executable
>fails, so it cannot return before the new executable has completed
>loading.  How does your code ensure this?  How is the load
>success/failure status passed back to the thread that calls "exec"?

**Synchronization Mechanism:**

1. **Semaphore Coordination**: I use `load_sema` in the `child_status` structure to coordinate between parent and child.

2. **Parent Blocking**: In `process_execute()`, after creating the child thread, the parent calls `sema_down(&child_status->load_sema)` to block until loading completes.

3. **Child Signaling**: In `start_process()`, after `load()` completes, the child sets `load_done = true` and `load_ok` to the result, then calls `sema_up(&child_status->load_sema)` to wake the parent.

4. **Status Transfer**: The parent checks `load_ok` after being awakened. If false, it cleans up the child_status and returns TID_ERROR.

**Code Flow:**
```c
// In process_execute():
tid = thread_create(...);
sema_down(&info->child_status->load_sema);  // Block until child loads
if (!info->child_status->load_ok)
  return TID_ERROR;  // Load failed
return tid;  // Load succeeded

// In start_process():
success = load(...);
info->child_status->load_ok = success;
sema_up(&info->child_status->load_sema);  // Wake parent
```

>B8: Consider parent process P with child process C.  How do you
>ensure proper synchronization and avoid race conditions when P
>calls wait(C) before C exits?  After C exits?  How do you ensure
>that all resources are freed in each case?  How about when P
>terminates without waiting, before C exits?  After C exits?  Are
>there any special cases?

**Synchronization Cases:**

1. **P waits before C exits:**
   - P finds C's child_status, sees `child_exited = false`
   - P blocks on `exit_sema` via `sema_down()`
   - When C exits, it sets `child_exited = true` and calls `sema_up(&exit_sema)`
   - P wakes up, gets exit code, cleans up child_status

2. **P waits after C exits:**
   - P finds child_status with `child_exited = true`
   - P skips semaphore wait, directly gets exit code and cleans up

3. **P terminates before C exits:**
   - P calls `child_status_cleanup()` which sets `parent_exited_first = true` on all children
   - When C later exits, it checks this flag and cleans up its own child_status
   - No memory leaks as C frees the record

4. **P terminates after C exits:**
   - Normal cleanup in `child_status_cleanup()` as child_status still exists
   - P frees all remaining child_status records

**Race Condition Prevention:**
- All child_status access is protected by the inherent atomicity of thread scheduling
- Semaphores provide proper synchronization between parent and child
- Boolean flags (`child_exited`, `parent_exited_first`) provide clear state tracking

**Resource Management:**
- `child_status` is heap-allocated and shared between parent and child
- Whichever process exits last is responsible for freeing the record
- No double-free issues due to careful flag management

#### RATIONALE

>B9: Why did you choose to implement access to user memory from the
>kernel in the way that you did?

I chose the **validate-then-access** approach over the **handle-page-faults** approach for several reasons:

1. **Simplicity**: The validation approach is much simpler to implement and debug. It uses existing Pintos functions (`is_user_vaddr`, `pagedir_get_page`) rather than requiring assembly code.

2. **Reliability**: Less prone to subtle bugs than assembly-based fault handling. The validation logic is explicit and easy to verify.

3. **Maintainability**: The code is more readable and easier to modify. Adding new validation checks is straightforward.

4. **Development Speed**: Faster to implement correctly, important for a learning project.

**Trade-offs:**
- **Performance**: Slower than optimized fault-handling approach, but adequate for Pintos
- **Elegance**: The fault-handling approach is more elegant but significantly more complex

For a production system, I might choose the fault-handling approach, but for Pintos, the validation approach provides the best balance of correctness, simplicity, and development efficiency.

>B10: What advantages or disadvantages can you see to your design
>for file descriptors?

**Advantages:**
1. **Simplicity**: Array-based lookup is O(1) and easy to implement
2. **Compatibility**: Matches standard Unix file descriptor behavior
3. **Automatic Cleanup**: Process exit automatically closes all files in the array
4. **Clear Ownership**: Each process owns its file descriptors independently

**Disadvantages:**
1. **Fixed Limit**: Hard limit of 128 file descriptors per process (FD_CNT)
2. **Memory Usage**: Array always allocates space for max FDs, even if unused
3. **Linear Search**: Finding free FD requires linear scan (though small array)
4. **No Sharing**: Processes can't share file descriptors (though this is by design)

**Potential Improvements:**
- Dynamic allocation using hash table or list for unlimited FDs
- Bitmap to track free FDs for faster allocation
- Reference counting for shared file descriptors
